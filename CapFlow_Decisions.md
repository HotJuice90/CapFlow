## Расчётное ядро

### 1. Налог — настраиваемый
Ставка налога и необлагаемый лимит задаются пользователем в настройках.
Движок применяет их к **годовому кумулятивному** процентному доходу по **всем**
активам (как точный НК РФ по структуре, но числа берутся из настроек, а не зашиты).
Фиксация налога — по итогам года (на 31 декабря) и в момент закрытия актива (Snapshot).

### 2. «Доход в день» — мгновенный
Единый KPI во всём приложении:

```
доход_в_день = текущая_сумма × ставка / дней_в_году
```

- День-каунт: `actual/365-366` (делим на реальное число дней в текущем году).
- Для накопительного счёта — капает на текущий остаток (как МТС/Яндекс Сейв).
- Для вклада без капы — постоянное число весь срок (мгновенный = усреднённый).
- Для вклада с капой — растёт по мере роста базы.

На главной и в аналитике «доход в день» = сумма по всем активам.

### 3. Капитализация — простой % по умолчанию + опция
- По умолчанию актив считается **простым процентом** (без капитализации).
- Капитализация — опциональный флаг + период (ежемес/ежекв/…). При включении движок
  капитализирует начисленные проценты в базу в дату выплаты.
- Режим «выплата процентов на сторону» в MVP фиксируется как `получено N` в истории
  актива (часть «уже заработано»), но **не копится** в отдельный баланс
  (свободный капитал вне MVP).

### 4. Свободный капитал — вне MVP
Убран с главной и из аналитики. На главной/аналитике остаётся только
**работающий капитал**. Вернётся вместе с поведением «Денежный» после MVP.

## Внешние данные

### 5. Курсы валют — API ЦБ РФ
Источник cbr.ru (бесплатный, официальный, раз в день, без ключей).
Офлайн — последний успешно полученный курс. Показываем источник + дату обновления.

### 6. Ключевая ставка ЦБ — авто из API ЦБ
Тянется автоматически (вместе с курсами). Нужна для «премии к ключевой» и налогового
лимита. Офлайн — последнее сохранённое значение.

### 7. Защита входа — нет в MVP
Биометрия/PIN не входят в первую версию. **Архитектурно** оставить место под
экран-замок (gate на старте навигации), чтобы вкрутить `expo-local-authentication`
позже без переписывания.

## История и аналитика

### 8. Snapshot — в обоих сценариях
И «Закрыть», и «Архивировать» создают Snapshot (фиксация итоговых цифр + версия
движка). **Но**: архивные снапшоты помечаются флагом `excludeFromAnalytics = true` —
доступны в истории, но не участвуют в расчётах доходности (архив = ошибочные/
неактуальные записи).

### 9. График капитала — реконструкция из первичных данных
История капитала строится задним числом из активов и их дат (открытие/закрытие/
пополнения), а не из ежедневных снимков. Ложится на принцип «храним только первичные
данные». Минус принят: редактирование актива задним числом перерисовывает прошлое.

### 10. Инсайты — 2-3 базовых правила
Детерминированные правила в MVP:
- «Скоро заканчивается крупный вклад» (срочные, осталось ≤ N дней).
- «Средняя ставка портфеля выше/ниже ключевой».
- (резерв) «Часть капитала освобождается в этом месяце».
Без сложного движка приоритизации — добавим позже.

### 11. Объясняющая аналитика + сравнение периодов — базово в MVP
Простые сравнения (сегодня↔вчера, месяц↔месяц) + объяснение по 1-2 значимым событиям.
**Архитектуру событий закладываем сразу** (создание/открытие/пополнение/закрытие/…),
даже если полный журнал — после MVP.

## Платформа и техстек

### 12. Платформа — Android-first
Основной фокус Android (iOS следом). Дизайн-гайд уже учитывает Android-грабли
(blur, шрифты, тени через `boxShadow`).

### 13. Язык — русский, i18n-ready
UI на русском, строки вынесены в словарь, чтобы добавить языки позже без переписывания.

### 14. Хранилище — AsyncStorage за слоем repository
AsyncStorage для MVP, но **строго за абстракцией `repository`**. Миграция на
`expo-sqlite` при росте (история событий, сотни активов) — без боли.

### 15. Первый запуск — демо-данные + сброс (обновлено)
На первом запуске наполняем приложение **демо-портфелем** (помечен `isDemo`), чтобы
срез читался на живых цифрах. В настройках — кнопка **«Удалить тестовые данные»**
(чистит только `isDemo`-записи). Пустые состояния по гайду (карточка + иконка +
подсказка + CTA) всё равно реализуем — они показываются после удаления демо.

## План сборки

### 16. Подход — вертикальный срез
Сначала одна сквозная фича целиком, потом расширяем вширь.

**Срез 1 (первый рабочий результат):**
1. Расчётный движок (чистый TS-модуль, без RN) + юнит-тесты — вклад (Срочный).
2. Минимальный справочник организаций (для создания актива).
3. Создание актива «Вклад» с живым предрасчётом.
4. Главная: доход в день (сумма) + работающий капитал + список активов.
5. Карточка вклада: Hero (доход/день, ставка, сумма, бейдж «осталось N дней») +
   финансовый результат (начислено/налог/чистыми) + второй блок (срок/прогресс).

Дальше: накопительный счёт (Бессрочный) → аналитика → календарь → конвертер → настройки.

### 17. Набор инструментов MVP
- **Вклад** (поведение: Срочный)
- **Накопительный счёт** (поведение: Бессрочный)
- **ЦФА — после MVP** (логика = вклад, добавится как тип над «Срочным»).

## Инженерные принципы (зафиксировано)

- **Расчётный движок — отдельный чистый модуль** `calc/` без зависимостей от RN:
  `calculate(asset, params, now) → derived`. Покрыт юнит-тестами. Единый источник
  истины для главной/аналитики/календаря/карточек.
- **Хранятся только первичные данные.** Все производные — на лету (по спеке).
- **День-каунт** везде `actual/365-366`.
- **Денежный формат** — российский: пробел-разделитель тысяч, символ валюты после
  числа через пробел (по спеке).
